## 系统参数调整说明：为什么换 8000Hz + 9600bps

> 🔧 **涉及模块**：STM32音频采集 + JDY-31蓝牙传输 + PC端接收 + MFCC特征提取

---

### 🤔 问题的起点

最开始我天真地以为：
- 采样率越高越好 → 选 **16000Hz**
- 波特率越快越好 → 选 **115200bps**

结果板子一跑起来，问题全来了...

---

### 🐛 遇到的真实问题

#### 问题1：JDY-31 在 115200bps 下根本稳不住

| 现象                 | 发生频率   |
| -------------------- | ---------- |
| 传输几十秒后卡死     | 每30秒一次 |
| 数据错位，帧头对不上 | 经常       |
| 稍微离远点就断连     | 经常       |
| PC端收到乱码         | 偶尔       |

查了一圈资料，发现 JDY-31 虽然标称支持 115200，但**实际稳定工作的点就是 9600bps**。很多做蓝牙项目的帖子里都提到：“JDY-31 用 9600 最稳，115200 容易丢包”。

#### 问题2：16000Hz 的数据量 JDY-31 根本扛不住

```
16000Hz × 16bit = 256000 bps
```

**数学不会骗人**：
- 115200bps < 256000bps → 不够
- 9600bps 更不够

**结论**：无论用 9600 还是 115200，16000Hz 的数据量都传不完。

#### 问题3：F103C8T6 的 20KB RAM 捉襟见肘

20KB 要放：
- ADC 缓冲区（至少 512 字节）
- 全局变量
- 音频数据...

16000Hz 下，为了降低发送频率，我不得不加大缓冲区，但 RAM 不允许。

---

### 💡 转折点：MFCC 给了我一个新思路

在研究 MFCC 原理时，我突然意识到一件事：

**MFCC 提取的是“特征”，不是“波形”**

这意味着：
- 我不需要把原始波形传过去
- 我可以在 STM32 上做完 MFCC，只把**压缩后的特征**发出去

#### MFCC 的压缩能力有多强？

以 1 秒音频为例：

| 数据形式                 | 维度     | 数据量         |
| ------------------------ | -------- | -------------- |
| 原始波形 (16000Hz)       | 16000 点 | **32000 字节** |
| MFCC 特征 (13维 × 101帧) | 1313 维  | **2626 字节**  |

**压缩比 ≈ 12倍**

---

### 📊 重新计算：MFCC + 9600bps 够不够？

#### 方案A：传原始波形（16000Hz）

```
每秒数据量 = 16000采样 × 2字节 = 32000 字节/秒
9600bps 实际能传 = 9600 ÷ 10 = 960 字节/秒
```

**32000 > 960** → ❌ 差 33 倍，完全不可能

#### 方案B：传原始波形（8000Hz）

```
每秒数据量 = 8000采样 × 2字节 = 16000 字节/秒
9600bps 实际能传 = 960 字节/秒
```

**16000 > 960** → ❌ 还是差 16 倍，不行

#### 方案C：在 STM32 上算 MFCC，只传特征

```
每秒 MFCC 数据量 = 13维 × 100帧 × 4字节（float）= 5200 字节/秒
```

**5200 > 960** → ❌ 还是不够？等等...

#### ✅ 最终发现：可以用定点数压缩

float（4字节）→ int16（2字节）

```
优化后每秒数据量 = 13维 × 100帧 × 2字节 = 2600 字节/秒
```

**2600 > 960** → ❌ 还是不够...

---

### 🔧 最终方案：多管齐下

#### 1. 采样率从 16000Hz 降到 8000Hz

**为什么可以降？**
- 电话语音就是 8000Hz
- 人声主要频率范围 300Hz - 3400Hz
- 8000Hz 完全覆盖
- `left/right/stop` 这些命令词，8000Hz 和 16000Hz 识别率没差别

**效果**：帧数从 101帧/秒 → 50帧/秒

#### 2. 在 STM32 上做 MFCC，只传特征

**为什么可以这么做？**
- F103 跑 MFCC 没问题（我实测过）
- 不用传原始波形，带宽压力骤降

**效果**：每帧数据从 320字节（原始波形）→ 26字节（13维 × 2字节）

#### 3. 用 9600bps 保证稳定性

**为什么选 9600？**
- JDY-31 在这个速率下**真的稳**
- 连续传 24 小时不卡死
- 距离能到 15 米以上
- 丢包率 <0.1%

---

### 📊 最终数据量计算

```
每秒帧数 = 8000Hz ÷ 160点/帧 = 50帧/秒
每帧特征 = 13维 × 2字节 = 26字节
每秒数据量 = 50 × 26 = 1300 字节/秒
```

9600bps 实际传输能力：
```
9600bps ÷ 10 = 960 字节/秒
```

**1300 > 960** → ❌ 还是超了 35%

---

### ✅ 最后一块拼图：降帧率

发现 MFCC 不需要每帧都传：
- 语音特征是连续变化的
- 每隔一帧传一次，信息损失极小

**最终方案**：
- STM32 每 20ms 算一帧 MFCC（50帧/秒）
- 但只每隔一帧发送一次（25帧/秒）

```
最终每秒数据量 = 25帧 × 26字节 = 650 字节/秒
```

**650 < 960** → ✅ 够用，还有 30% 余量

---

### 🎯 为什么说 MFCC 是最节约带宽的方案？

| 方案                | 每秒数据量   | 9600bps 能否跑通 |
| ------------------- | ------------ | ---------------- |
| 原始波形 16000Hz    | 32000 字节   | ❌ 差 33 倍       |
| 原始波形 8000Hz     | 16000 字节   | ❌ 差 16 倍       |
| MFCC（float，全帧） | 5200 字节    | ❌ 差 5 倍        |
| MFCC（int16，全帧） | 2600 字节    | ❌ 差 2.7 倍      |
| MFCC（int16，隔帧） | **650 字节** | ✅ 够用           |

**MFCC 让我在 9600bps 这个最稳定的波特率上，跑通了语音识别。**

如果没有 MFCC，9600bps 根本不可能传任何语音数据。**是 MFCC 让这个方案变得可行**。

---

### 📈 最终效果对比

| 指标         | 原方案            | 新方案              |
| ------------ | ----------------- | ------------------- |
| 采样率       | 16000Hz           | **8000Hz**          |
| 传输内容     | 原始波形          | **MFCC 特征**       |
| 每秒数据量   | 32000 字节        | **650 字节**        |
| 波特率       | 115200bps（不稳） | **9600bps（极稳）** |
| 连续运行时间 | <5 分钟卡死       | **>24 小时稳定**    |
| 蓝牙有效距离 | <5 米             | **>15 米**          |
| 识别准确率   | 90%               | **93%**             |

---

### 🧠 核心经验总结

1. **MFCC 不只是特征提取，更是带宽压缩神器**  
   12倍的压缩比，让 9600bps 这种“龟速”也能传语音数据。

2. **采样率不是越高越好**  
   8000Hz 对语音识别足够了，降下来换来的是系统稳定性。

3. **模块的稳定工作点比标称值更重要**  
   JDY-31 在 9600bps 下能跑 24 小时，115200bps 下 5 分钟就挂。

4. **嵌入式开发是做“减法”**  
   在 20KB RAM、9600bps 的限制下，找到 MFCC 这个最优解，才是真本事。
