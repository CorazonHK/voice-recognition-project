> 📅 **版本**：v1.0  
> 🔧 **主控**：STM32F103C8T6

> 🎯 **用途**：快速验证硬件方案 + 算法可行性

### 🎯 这个版本的目标

F103 版本从一开始就不是“最终产品”，它的定位很明确：

| 目标           | 说明                                |
| -------------- | ----------------------------------- |
| ✅ 快速验证硬件 | MAX9814、HC-06、OLED 能不能正常工作 |
| ✅ 打通通信链路 | 音频数据能不能从 STM32 传到 PC      |
| ✅ 验证算法     | MFCC + CNN 在 PC 端能不能跑通       |
| ✅ 采集真实数据 | 用这套硬件录一批自己的语音          |
| ❌ 不是最终架构 | 不上 FreeRTOS，不用复杂架构         |

**一句话**：先让东西跑起来，再考虑怎么跑得好。

### 🧠 学到了什么

#### 硬件方面的经验

1. **以实物为准**  
   MAX9814 的引脚排序让我长记性了：永远不要相信网上的引脚图，拿到模块先拿万用表打一遍。

2. **电源线要粗**  
   第一版板子电源线太细，蓝牙一发射电压就掉。后来加粗到 27mil，问题解决。

3. **电容要贴芯片放**  
   去耦电容放远了等于白放。0.1μF 必须紧贴芯片引脚，这个坑我踩过。

#### 软件方面的经验

1. **裸机够用就别急着上系统**  
   F103 版本没用 FreeRTOS，少了很多调试的麻烦。先把功能跑通，上系统的事留给 F407。

2. **帧头同步很重要**  
   一开始没加帧头，PC 端经常收错数据。加上 `AA 55` 之后，再也没出过问题。

3. **数据分包比降低速率更有效**  
   从一次发 1024 字节改成每 20ms 发 320 字节，传输稳定性大幅提升。

#### 算法方面的经验

1. **MFCC 的参数不能错**  
   hop_length、n_mfcc、n_fft 这些参数，训练和推理必须完全一致，否则识别率直接归零。

2. **CMSIS-DSP 有坑**  
   arm_bitreversal_32 链接不上的问题，最后自己手写 fix 才解决。官方库不一定靠谱。

---

### ⚠️ 这个版本的局限

1. **裸机架构，代码耦合度高**  
   所有功能都挤在 main 循环里，加新功能越来越难。

2. **没有用 FreeRTOS**  
   任务管理靠轮询，实时性一般，CPU 利用率不高。

3. **F103 的 RAM 太小**  
   20KB 要放 ADC 缓冲区、全局变量、栈，有点捉襟见肘。

4. **采样率只有 8000Hz**  
   够用，但想提高识别率的话，还是 16000Hz 更好。

5. **蓝牙还是 9600bps**  
   稳是稳，但想传更高质量的数据，带宽不够。
